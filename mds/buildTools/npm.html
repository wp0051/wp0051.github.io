执行流程

环境变量找到npm并调用执行

  • 终端从环境变量中获取npm(node.js安装完成之后，环境变量添加node和npm(node的一个脚本))
  • 当使用 npm run 执行脚本时，npm 会设置一些环境变量。其中，npm_config_argv 是一个环境变量，它包含了 npm 运行脚本时的命令行参数，比如上面的--account。
  • 这个环境变量是由 npm 自动设置的，所以我们在脚本中可以直接通过 process.env.npm_config_argv 访问，而不需要自己声明。
  • 实际上，npm_config_argv 是一个字符串，内容是 JSON 对象，它包含了原始的命令行参数。例如，当你运行 npm run dev --account，那么 npm_config_argv 可能看起来像这样：
{
  "remain": ["dev"],
  "cooked": ["run", "dev", "--account"],
  "original": ["run", "dev", "--account"]
}
  • 但是从 npm 7 开始，这个环境变量不再自动设置。取而代之的是，你可以通过 process.env.npm_config_* 来获取通过 -- 传递的参数，或者使用 process.argv 来获取参数,但是需要按照npm run dev -- --account的方式传递

读取dev的srcipt并生成队列

  • 执行run dev，读取当前目录下的package.json文件下的scripts.dev的值"cross-env NODE_ENV=development && gulp dev && node build/dev-server.js"
{
    "scripts": {
        "dev": "cross-env NODE_ENV=development && gulp dev && node build/dev-server.js",
        "build": "cross-env NODE_ENV=production && node build/build.js && gulp build",
    }
}
  • npm 会启动一个子 shell，执行这个命令，这个命令字符串由三部分组成，用 && 连接，表示按顺序执行，每个命令（由&&连接）都会等待前一个命令完成（根据退出码判断成功与否）再执行下一个，非0退出码通常表示失败。
  • 当子shell中的整个命令序列执行完毕，子shell进程会退出，并且将退出码返回给npm进程。
  • 最后一个命令（例如：node build/dev-server.js）如果是一个长期运行的服务器进程，它不会退出，那么子shell就不会退出，npm进程就会一直等待，直到我们收到结束进程（ctrl + c）

cross-env NODE_ENV=development

  • cross-env NODE_ENV=development：cross-env 是一个跨平台设置环境变量的工具，这段就是在当前进程中将NODE_ENV设置为development

gulp dev

  • gulp dev：gulp也是需要从环境变量中去取的，但是它是什么时候变成环境变量的呢？
    • 在我们项目执行npm install gulp --save-dev，将gulp作为开发依赖下载执行的时候，他会存储在node_modules/.bin目录下
    • 在npm run执行时，会整体将node_modules/.bin添加到环境变量中去
    • gulp dev会在当前目录下执行gulpfile.js文件，并且执行定义的task任务
    // 启动开发环境npm run dev
gulp.task('dev', ['clean:static'], function (cb) {
    sequence(['copy:devimage','copy:font', 'copy:devmoduleImage', 'copy:devconf', 'copy:JS'], ['commonCss:dev', 'modulesCss:devseparate'], cb)
    })
    • gulp在判断执行完任务后返回退出code，npm当前进程执行下一个命令

node build/dev-server.js

  • node build/dev-server.js，node执行dev-server.js文件，执行webpack打包，启动本地开发服务器；
